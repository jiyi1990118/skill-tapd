# 需求分析专业规范

## 一、需求理解六维度（详细）

### 1. 业务目标（Why）

**必须明确**：
- 这个需求要解决什么业务问题？
- 预期带来什么业务价值？（可量化的 KPI）
- 如何衡量成功？

**提问模板**：
```
- 业务目标是什么？
- 这个需求的价值是什么？
- 成功的标准是什么？有具体指标吗？
- 不做这个需求会有什么影响？
```

**示例**：
- ❌ "做一个用户登录功能"
- ✅ "做一个用户登录功能，目标是提高用户留存率（从 60% 提升到 75%），通过减少注册摩擦和支持第三方登录"

---

### 2. 用户角色（Who）

**必须明确**：
- 谁会使用这个功能？
- 不同角色的需求有何不同？
- 各角色的权限边界？

**提问模板**：
```
- 哪些用户会使用此功能？
- 不同角色的使用场景有何差异？
- 各角色的权限如何划分？
```

**示例**：
- 普通用户：快速登录，支持手机号/邮箱
- 管理员：后台管理用户，查看登录日志
- 游客：部分功能可免登录体验

---

### 3. 使用场景（When & Where）

**必须明确**：
- 用户在什么情况下会使用？
- 有哪些典型使用路径？
- 前置条件和后置条件是什么？

**提问模板**：
```
- 用户在什么情况下会使用此功能？
- 典型的使用流程是什么？
- 使用前需要什么前置条件？
- 使用后会产生什么结果？
```

**示例**：
- 场景1：首次访问，需要注册
- 场景2：老用户回访，快速登录
- 场景3：忘记密码，需要找回
- 场景4：多设备同步登录状态

---

### 4. 业务价值（Value）

**必须明确**：
- 为什么现在做？优先级依据是什么？
- 不做会有什么影响？
- 与其他需求的优先级对比？

**提问模板**：
```
- 为什么现在要做这个需求？
- 优先级为什么是 high/medium/low？
- 如果延后会有什么影响？
```

**示例**：
- 高优：影响核心流程，阻塞其他需求
- 中优：改善用户体验，有竞品压力
- 低优：锦上添花，可延后

---

### 5. 边界条件（Boundary）

**必须明确**：
- 哪些在范围内？哪些不在？
- 有哪些约束条件（技术、合规、预算）？
- 与现有系统的集成点？

**提问模板**：
```
- 此需求的范围是什么？哪些在内？哪些在外？
- 有哪些约束条件？
- 需要与哪些系统集成？
- 有哪些依赖项？
```

**示例**：
- 范围内：手机号登录、微信登录、密码找回
- 范围外：指纹登录、人脸识别（留待 v2.0）
- 约束：必须符合网络安全法，用户数据加密存储

---

### 6. 成功标准（Done）

**必须明确**：
- 如何判断需求已完成？
- 验收标准是什么？
- 性能、安全、体验的具体要求？

**提问模板**：
```
- 如何判断此需求已完成？
- 验收标准是什么？
- 功能、性能、安全、体验各有什么具体要求？
```

**示例**：
- 功能验收：用户可以通过手机号+验证码登录
- 性能验收：登录响应时间 < 2s
- 数据验收：用户信息正确存储，无数据泄露
- 兼容性验收：支持 iOS/Android/Web 三端

## 三、需求拆解方法（金字塔结构）

### 拆解层级

```
Level 1: 业务目标 (Business Goal)
  ↓
Level 2: 用户故事 (User Story) → TAPD Story
  ↓
Level 3: 功能模块 (Feature Module)
  ↓
Level 4: 功能点 (Feature)
  ↓
Level 5: 开发任务 (Dev Task) → TAPD Task
  ↓
Level 6: 测试任务 (Test Task) → TAPD Task
  ↓
Level 7: 验收项 (Acceptance Criteria)
```

### 拆解原则

1. **单一职责** - 每个任务只做一件事
2. **可测试** - 每个任务都有明确的验收标准
3. **可估算** - 单个任务工作量 ≤ 2 天（超过则继续拆解）
4. **依赖明确** - 标注任务间的依赖关系
5. **完整覆盖** - 前端+后端+测试，缺一不可

### 拆解示例

**业务目标**：提高用户留存率，从 60% 提升到 75%

**用户故事（Story）**：
```
[Story-101] 作为新用户，我希望可以快速注册账号，以便使用平台服务
[Story-102] 作为老用户，我希望可以快速登录，以便继续之前的操作
[Story-103] 作为忘记密码的用户，我希望可以通过手机找回密码
```

**功能模块（Story-102 拆解）**：
- 模块1：用户注册
- 模块2：用户登录
- 模块3：密码管理

**功能点（模块2 拆解）**：
- 功能点1：手机号+密码登录
- 功能点2：手机号+验证码登录
- 功能点3：微信一键登录
- 功能点4：记住登录状态

**开发任务（功能点1 拆解）**：
```
[Task-201] 后端：实现登录接口（/api/login）- 1天
[Task-202] 后端：密码加密和校验逻辑 - 0.5天
[Task-203] 后端：生成 JWT Token - 0.5天
[Task-204] 前端：登录页面 UI - 1天
[Task-205] 前端：表单校验和错误提示 - 0.5天
[Task-206] 前端：调用登录接口并存储 Token - 0.5天
```

**测试任务**：
```
[Task-207] 测试：功能测试用例编写 - 0.5天
[Task-208] 测试：接口自动化测试 - 1天
[Task-209] 测试：前端 E2E 测试 - 0.5天
```

**验收项（功能点1）**：
```
- [ ] 用户输入正确的手机号和密码，可以登录成功
- [ ] 用户输入错误的密码，显示"密码错误"提示
- [ ] 用户连续输入错误密码 5 次，账号锁定 30 分钟
- [ ] 登录成功后，Token 存储在 localStorage
- [ ] 登录响应时间 < 2s（P95）
```

## 四、验收标准设计（Given/When/Then）

### 标准格式

```gherkin
Given [前置条件]
When [操作]
Then [预期结果]
And [附加结果]
```

### 功能验收标准

```gherkin
Scenario: 正常登录
  Given 用户已注册且未登录
  When 用户输入正确的手机号和密码
  And 点击"登录"按钮
  Then 系统验证通过
  And 跳转到首页
  And 显示用户昵称
```

### 数据验收标准

```gherkin
Scenario: Token 生成
  Given 用户登录成功
  When 系统生成 JWT Token
  Then Token 包含 user_id、role、expire_time 字段
  And Token 有效期为 7 天
  And Token 存储在数据库 user_sessions 表
```

### 权限验收标准

```gherkin
Scenario: 权限验证
  Given 用户角色为"普通用户"
  When 用户尝试访问管理后台
  Then 系统返回 403 Forbidden
  And 显示提示"无权限访问"
```

### 异常场景验证

```gherkin
Scenario: 账号锁定
  Given 用户已输入错误密码 4 次
  When 用户第 5 次输入错误密码
  Then 系统锁定账号 30 分钟
  And 显示提示"密码错误次数过多，账号已锁定 30 分钟"
  And 记录安全日志
```

### 性能验证

```gherkin
Scenario: 响应时间
  Given 系统并发用户数 1000
  When 用户点击"登录"按钮
  Then 登录接口响应时间 P95 < 2s
  And 登录接口响应时间 P99 < 5s
  And 系统 CPU 使用率 < 70%
```

## 七、常见错误与纠正

### ❌ 错误 1：需求描述过于简单
**错误示例**：
```
做一个用户登录功能
```

**纠正**：
```
需求信息过于简单，请补充：
- 业务目标是什么？
- 支持哪些登录方式？
- 验收标准是什么？
```

---

### ❌ 错误 2：验收标准模糊
**错误示例**：
```
用户可以登录
```

**纠正**：
```
验收标准不够明确，请改为：

Given 用户已注册且未登录
When 用户输入正确的手机号和密码，点击"登录"
Then 系统验证通过，跳转到首页，显示用户昵称
```

---

### ❌ 错误 3：任务粒度过大
**错误示例**：
```
[Task-201] 实现用户登录模块 - 5天
```

**纠正**：
```
任务粒度过大（> 2天），需继续拆解：

[Task-201] 后端：实现登录接口 - 1天
[Task-202] 后端：密码加密校验 - 0.5天
[Task-203] 前端：登录页面 UI - 1天
[Task-204] 前端：表单校验 - 0.5天
[Task-205] 测试：功能测试 - 1天
```
